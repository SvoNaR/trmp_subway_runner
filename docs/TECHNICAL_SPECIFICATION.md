# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## на разработку мобильного приложения "Subway Runner"

---

## 1. ВВЕДЕНИЕ

### 1.1 Цели
Разработка мобильного приложения в жанре endless runner (бесконечный раннер) по типу игры Subway Surfers для платформ Android и iOS с использованием кроссплатформенных технологий.

### 1.2 Соглашения о терминах
- **Endless Runner** - жанр игр, где персонаж автоматически бежит вперед бесконечно
- **Lane** - полоса движения (в игре 3 полосы)
- **Obstacle** - препятствие, которое нужно обойти
- **Coin** - монета для сбора
- **Score** - счет игрока
- **Leaderboard** - таблица лидеров

### 1.3 Предполагаемая аудитория
- Пользователи мобильных устройств Android и iOS
- Любители аркадных игр
- Возраст: 6+

### 1.4 Масштаб проекта
- Мобильное приложение с клиент-серверной архитектурой
- Серверная часть на Node.js
- Клиентская часть на React Native (Expo)
- База данных SQLite
- Система аутентификации пользователей
- Таблица лидеров онлайн

### 1.5 Ссылки на источники
1. Официальная документация React Native: https://reactnative.dev
2. Документация Expo: https://docs.expo.dev
3. Документация Node.js: https://nodejs.org/docs
4. Спецификация SRS IEEE 830-1998

## 2. ОБЩЕЕ ОПИСАНИЕ

### 2.1 Видение продукта
Создание увлекательной аркадной игры в стиле Subway Surfers с простым управлением, ярким дизайном и системой соревнований между игроками через онлайн-таблицу лидеров.

### 2.2 Функциональность продукта
- Регистрация и авторизация пользователей
- Игровой процесс с тремя полосами движения
- Сбор монет и избегание препятствий
- Подсчет очков и сохранение результатов
- Онлайн-таблица лидеров
- Профиль игрока со статистикой

### 2.3 Классы и характеристики пользователей
- **Казуальные игроки** - играют от случая к случаю
- **Хардкорные игроки** - стремятся к высоким результатам
- **Возраст**: от 6 лет и старше
- **Технические навыки**: базовое владение смартфоном

### 2.4 Среда функционирования продукта
- **Операционные системы**: Android 5.0+, iOS 11.0+
- **Сервер**: Node.js 14+, любая ОС (Windows, Linux, macOS)
- **База данных**: SQLite 3.x
- **Сеть**: Требуется подключение к интернету для сохранения результатов

### 2.5 Рамки, ограничения, правила и стандарты
- Разработка ведется на React Native с использованием Expo
- Серверная часть на Express.js
- REST API для взаимодействия клиент-сервер
- JWT токены для аутентификации
- Кодировка UTF-8 для поддержки кириллицы

### 2.6 Документация для пользователей
- Встроенные подсказки в игре
- Экран обучения при первом запуске
- Полная техническая документация

### 2.7 Допущения и зависимости
- Пользователь имеет стабильное интернет-соединение
- Устройство поддерживает мультитач для свайпов
- Сервер доступен по локальной сети или интернету

## 3. ФУНКЦИОНАЛЬНОСТЬ СИСТЕМЫ

### 3.1 Функциональный блок 1: Аутентификация пользователей

#### 3.1.1 Описание и приоритет
Регистрация новых пользователей и вход существующих.  
**Приоритет**: Высокий

#### 3.1.2 Причинно-следственные связи
- Регистрация → Создание учетной записи → Вход в систему
- Вход → Проверка credentials → Предоставление доступа

#### 3.1.3 Функциональные требования
- FR-1.1: Система должна позволять регистрацию с email и паролем
- FR-1.2: Система должна проверять уникальность email
- FR-1.3: Система должна хешировать пароли перед сохранением
- FR-1.4: Система должна выдавать JWT токен после входа
- FR-1.5: Система должна проверять валидность токена

### 3.2 Функциональный блок 2: Игровой процесс

#### 3.2.1 Описание и приоритет
Основная игра с управлением персонажем, препятствиями и монетами.  
**Приоритет**: Критический

#### 3.2.2 Алгоритмы и workflows
```
Старт игры → Генерация препятствий → Движение персонажа
→ Обнаружение столкновений → Подсчет очков → Конец игры
→ Сохранение результата
```

#### 3.2.3 Функциональные требования
- FR-2.1: Система должна отображать 3 полосы движения
- FR-2.2: Система должна реагировать на свайпы влево/вправо
- FR-2.3: Система должна генерировать препятствия случайным образом
- FR-2.4: Система должна генерировать монеты для сбора
- FR-2.5: Система должна фиксировать столкновения с препятствиями
- FR-2.6: Система должна увеличивать скорость постепенно
- FR-2.7: Система должна подсчитывать очки и монеты

### 3.3 Функциональный блок 3: Таблица лидеров

#### 3.3.1 Описание и приоритет
Онлайн-рейтинг игроков по очкам.  
**Приоритет**: Средний

#### 3.3.2 Функциональные требования
- FR-3.1: Система должна сохранять результаты игр
- FR-3.2: Система должна отображать топ-10 игроков
- FR-3.3: Система должна сортировать по максимальному счету
- FR-3.4: Система должна обновляться в реальном времени

### 3.4 Функциональный блок 4: Профиль пользователя

#### 3.4.1 Описание и приоритет
Личный кабинет с статистикой игрока.  
**Приоритет**: Средний

#### 3.4.2 Функциональные требования
- FR-4.1: Система должна отображать имя пользователя
- FR-4.2: Система должна показывать статистику игр
- FR-4.3: Система должна позволять выйти из аккаунта

## 4. ТРЕБОВАНИЯ К ВНЕШНИМ ИНТЕРФЕЙСАМ

### 4.1 Интерфейсы пользователя (UX)
- Яркий, красочный дизайн
- Интуитивное управление свайпами
- Плавные анимации перемещений
- Обратная связь при сборе монет и столкновениях
- Адаптивный интерфейс под разные экраны

### 4.2 Программные интерфейсы (API)

#### POST /api/register
Регистрация пользователя  
**Тело запроса:**
```json
{
  "username": "string",
  "email": "string",
  "password": "string"
}
```

#### POST /api/login
Вход в систему  
**Тело запроса:**
```json
{
  "email": "string",
  "password": "string"
}
```

#### POST /api/scores
Сохранение результата  
**Заголовки:** Authorization: Bearer {token}  
**Тело запроса:**
```json
{
  "score": number,
  "coins": number,
  "distance": number
}
```

#### GET /api/leaderboard
Получение таблицы лидеров

#### GET /api/progress
Получение прогресса пользователя

### 4.3 Интерфейсы оборудования
- Сенсорный экран для управления свайпами
- Виброотклик (опционально)

### 4.4 Интерфейсы связи и коммуникации
- HTTP/HTTPS протокол
- JSON формат данных
- WebSocket (потенциально для мультиплеера)

## 6. ПРИЛОЖЕНИЯ

### Приложение А: Глоссарий
- **JWT** - JSON Web Token, стандарт для аутентификации
- **REST** - архитектурный стиль построения API
- **SQLite** - реляционная база данных
- **React Native** - фреймворк для мобильной разработки
- **Expo** - платформа для упрощения разработки на React Native

### Приложение Б: Диаграммы

#### Диаграмма прецедентов
```
[Игрок] --> (Регистрация)
[Игрок] --> (Вход)
[Игрок] --> (Играть)
[Игрок] --> (Просмотр лидеров)
[Игрок] --> (Просмотр профиля)
```

#### Схема базы данных
```
USERS (id, username, email, password, created_at)
GAME_SCORES (id, user_id, score, coins, distance, created_at)
USER_PROGRESS (id, user_id, level, total_coins, unlocked_characters, last_played)
```
